Lesson notes - Model access

Model access is useful to restrict what other models can ref a given model.

  • This is separate from database or user access - this doesn’t do anything with database permissions, this is purely a dbt configuration

The main use case of this is managing dependencies. If you have a huge dbt project (or multi dbt project) setup, then if all of your data models in your department are used elsewhere it can be a pain to maintain dependencies. By restricting which of your dbt models can be used, and where, it can help avoid unwanted downstream dependencies.

  • Let’s say you maintain 30 dbt models in your Sales data team, and only want to let 5 of these be used by Marketing / Finance etc. - then model access is your friend!

Access

There are 3 levels of access:

  • Private: only other models in the same group can ref this model
  • Protected (default): only other models in the same group or dbt project can ref this model
  • Public: any other model - in the same group, project, or separate dbt project can ref this model
    • e.g. If you tried to ref a model in dbt_packages/dbt_utils (there aren’t any, but just for this example let’s say there are) that wasn’t set to public access you would get an error

Access is defined at a model level, and currently cannot be defined on a project or folder level.

So what’s a group?

A dbt group is a YML defined concept of a set of owners - such as Sales, Finance, or Marketing. Like everything else in dbt, it’s defined in a YML file:

groups:
  - name: sales
    owner:
      # 'name' or 'email' is required; additional properties allowed
      email: sales@my-company.com
      slack: sales-data
      github: sales-data-team

This can be in a separate file from your models, or in the same YML file (same as everything else in dbt - the structure of the files doesn’t matter). In a large dbt project it would be sensible to have at least 1 file dedicated to defining groups.

Groups can be defined:

  • …on a project level in dbt_project.yml
models:
    marts:
	      sales:
					  +group: sales
				marketing:
					  +group: marketing
  • …on a model level in the YML file
models:
  - name: orders
    description: "Table of order level information"
    config:
			# NOTE: +group won't work - you need to use just group!
      group: sales
  • …or on a model level in a SQL file
{{ config(group = 'marketing') }}

File you need to update:

models/marts/orders.yml I’ve not written out the full file here - just the bits that need changing. As mentioned in the video, you could (and probably in practice should) store your groups in a separate YML file - e.g. groups.yml. 

groups:
  - name: sales
    owner:
      email: sales@my-company.com

	# Optional, this is just to illustrate
  - name: marketing
    owner:
      email: marketing@my-company.com

models:
  - name: orders
    description: "Table of order level information"
    config:
      group: sales

    access: private

To test this works - you can create a separate model (under the models folder or dbt won’t recognise it as a model!), and do a simple SELECT * FROM {{ ref('orders') }} and either leave the group blank or set {{ config(group = 'marketing') }} at the top of the file. When you run this new model you should get an error.
