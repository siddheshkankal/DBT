Lesson notes - Model versions

Model versions are a way of having multiple versions of a single dbt model in a dbt project at once. This is useful when you want to make potentially breaking changes to a model, but want to not force all downstream users to immediately handle the breaking changes.

You can introduce a new version of a dbt model by simply creating a new file with modelname_vN.sql where N is the model version.

For example, we could have:

  • stg_ecommerce__products_v1.sql (old version)
  • stg_ecommerce__products.sql (latest release)

Then, in the YML for the model you will need to specify the model versions and what changes they’ve made.

models:
  - name: stg_ecommerce__products

		columns:
			...

    latest_version: 2
    versions:
    
      - v: 1
				columns:
          # Brand column introduced in v2
          - include: all
            exclude: [brand]
    
      - v: 2
				# We could leave this blank, this is just to illustrate
        columns:
          - include: all

You can specify the columns manually, rather than the include/exclude logic:

models:
  - name: stg_ecommerce__products
    latest_version: 1
    versions:
    
      - v: 1
				# Note: I've excluded the tests/descriptions here to keep it short
				# But you still need to put them in!
				columns:
          - name: product_id
          - name: cost
          - name: retail_price
          - name: department
    
      - v: 2
				columns:
          - name: product_id
          - name: cost
          - name: retail_price
          - name: department
					- name: brand

The include/exclude route highlighting just the changes is preferable as it’s a lot easier to track what’s changed.

Another personal preference is making sure that the latest version of the model is always saved in the unversioned file name, and kept in the same database location.

This is done with the alias config:

			- v: 2
        columns:
          - include: all
        # Makes this table stays as stg_ecommerce__products in our database!
        config:
          alias: stg_ecommerce__products

I prefer this for 2 reasons:

  1. BI tools like Looker / Tableau / Power BI typically will look for a fixed table name. You can achieve this using a view - but I find it simpler to always make sure that the latest version of the model is saved under schema.model_name rather than schema.model_name_vN.
  1. It’s always clear which SQL file contains the latest version, and in the git history (tracked changes) you can see the changes between versions rather than a brand new file being created each time

NOTE: to keep an unversioned model name, like stg_ecommerce__products.sql, you need to make sure you do the alias above!

When you run a versioned model, it will create a table (or view etc.) in your database for every version of your model. In the example above, it would create both schema.stg_ecommerce__products_v1 and schema.stg_ecommerce__products.

When you {{ ref() }} a model, if you don’t specify the version then dbt will always point at the latest_version set. If there’s a newer version of the model available that’s not marked as the latest release then dbt will still run but will give you a warning:

You can also force dbt to select a given version as you can see above. In our case, we could do {{ ref('stg_ecommerce__products', v='1') }}.

Files you need to create/update:

models/staging/old_versions.stg_ecommerce__products_v1.sql

This should just be the old version of the model copied into a new file.

WITH source AS (
	SELECT *

	FROM {{ source('thelook_ecommerce', 'products') }}
)

SELECT
	-- IDs
	id AS product_id,

	-- Other columns
	cost,
	retail_price,
	department

FROM source

models/staging/old_versions.stg_ecommerce__products.sql

This adds the brand column. NOTE: for this to work, you must alias the version 2 data model as stg_ecommerce__products - otherwise you’ll need to call this stg_ecommerce__products_v2.sql.

  • I personally prefer always having the current version of the model kept under the unversioned name (i.e. no _v suffix) as it always makes it clear which version of the model is current.
WITH source AS (
	SELECT *

	FROM {{ source('thelook_ecommerce', 'products') }}
)

SELECT
	-- IDs
	id AS product_id,

	-- Other columns
	cost,
	retail_price,
	department,
	brand -- new to version 2

FROM source

models/staging/stg_ecommerce__products.sql

version: 2

models:
  - name: stg_ecommerce__products
    description: "Table of products on the online store"
    columns:
      - name: product_id
        tests:
          - primary_key:
              severity: error

      - name: cost
        description: "How much the product cost the business to purchase"
        tests:
          - not_null
          - dbt_expectations.expect_column_values_to_be_between:
              min_value: 0

      - name: retail_price
        description: "How much the product retails for on the online store"
        tests:
          - not_null
          - dbt_expectations.expect_column_values_to_be_between:
              min_value: 0
          - dbt_utils.expression_is_true:
              expression: ">= cost"

      - name: department
        description: "Whether the item is Menswear or Womenswear"
        tests:
          - not_null
          - accepted_values:
              values:
                - "Men"
                - "Women"
              # quote defaults to true, but this just shows how you could set it to
              # false if you wanted to have non-string values (e.g. numbers or a TRUE/FALSE column)
              quote: true

      - name: brand


    latest_version: 2
    versions:
      - v: 1
        columns:
          - include: all
            exclude: [brand]

      - v: 2
        columns:
          - include: all
        # Makes this table stays as stg_ecommerce__products in our database!
        config:
          alias: stg_ecommerce__products

If you wanted to instead make version 1, without the brand column, the latest version that {{ ref ('stg_ecommerce__products') }} points to you need to:

  • Change latest_version above to be 1
  • Change the file names for:
    • stg_ecommerce__products_v1 → stg_ecommerce__products
    • stg_ecommerce__products → stg_ecommerce__products_v2
  • Set the alias for v: 1 to be the unversioned model name
    				config:
          alias: stg_ecommerce__products
