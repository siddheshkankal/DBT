Lesson notes - using different profiles

dbt, by default will look in the following order for a profiles.yml file (read more here):

  1. Where you manually specify using --profiles-dir
  1. Where you set the DBT_PROFILES_DIR environment variable
  1. The current working directory
  1. ~/.dbt

Remember, we set the DBT_PROFILES_DIR variable in our settings.json file in VSCode:

		"terminal.integrated.env.osx": {
        "DBT_PROFILES_DIR": "~/.dbt",

We did this to make sure that the dbt Power User extension would work! Remember we can use dbt debug --config dir to check where dbt is pulling our default profiles.yml file from.

If we wanted to change the file we were pulling from, we could use this command below. To get this working, you’ll need to create a file dbt_cloud_profiles/profiles.yml (see lower down in this doc for an example).

dbt --profiles-dir ./dbt_cloud_profiles run -s stg_ecommerce__products

This type of setup is most common in production deployments of dbt, where you want to use 1 centralised profile for all of your runs.

If we wanted to change the target profile we use from a profiles.yml, we could use this command below. Again, to get this working you’ll need to change your own ~/.dbt/profiles.yml file to add a new target called “testing”.

dbt run -s stg_ecommerce__products —-target testing

Personally, I only ever have 1 target profile in my own ~/.dbt/profiles.yml. I only ever use different target profiles within a profiles.yml file stored in the project itself (as per the dbt_cloud_profiles folder above) - 1 profile for testing code changes on GitHub, 1 profile for testing a critical part of our pipeline, and 1 for a full production run, each with their own separate schemas.

One main point to takeaway here is that regardless of which profiles.yml file you use, they all must have a key at the top of lessons: to tie back to the profile: key in your dbt_project.yml file.

# This setting configures which "profile" dbt uses for this project.
profile: 'lessons'

You can try creating this file, but it’s not needed for the project - just testing this functionality

dbt_cloud_profiles/profiles.yml

lessons:
  target: production
  outputs:
    production:
      dataset: dbt_production
      job_execution_timeout_seconds: 300
      job_retries: 1
      location: US
      method: oauth
      priority: interactive
      project: [Your project here!]
      threads: 60
      type: bigquery

As above, you can update your own profiles.yml file to add additional targets

~/.dbt/profiles.yml

lessons:
  target: dev
  outputs:
    dev:
      dataset: dbt_dev
      job_execution_timeout_seconds: 300
      job_retries: 1
      location: US
      method: oauth
      priority: interactive
      project: [Your project here!]
      threads: 60
      type: bigquery

    testing:
      dataset: dbt_test
      job_execution_timeout_seconds: 300
      job_retries: 1
      location: US
      method: oauth
      priority: interactive
      project: [Your project here!]
      threads: 60
      type: bigquery
