Lesson notes - first mart SQL model

Your marts folder will typically be divided into subfolders based on the end users - e.g. core_models, finance, sales etc.

Another difference with these models is that you’ll typically follow a different naming convention to your staging and intermediate models.

1️⃣ the dimension (dim) and fact (fct) approach

This approach involves prefixing files with:

  • dim_ if it’s one row per “thing”, such as an order or a product, that can change over time - an order can change status, a product can change department, etc.
  • fct_ if it’s one row per “thing that happened”, such as a click on a website, or a daily statistic for a user (fct_users_daily) - that won’t change over time

In practise, the naming of your models won’t make any difference to functionality - it’s more an easy way to organise your data models.

  • Another thing to mention here is that you don’t use the double underscore __ in the SQL model name as these tables typically aren’t associated with a specific data source - and calling a table dim_ecommerce__orders might be unnecessarily complicated if you only have 1 data source for orders!

A benefit of this approach is as above - it’s easy to organise your data, but a drawback is that if you’re surfacing these tables to non-technical end users then honestly it can be quite confusing.

2️⃣ the simpler approach

Simpler doesn’t always been better, but the second approach is to omit the prefixes entirely, e.g. opting for orders instead of dim_orders, users_daily for a table of 1 row per user per day (a fact table summarising information about that user on that day), and web_clicks.

A nice benefit of this approach is that tables relating to the same entities are grouped together, e.g. users and users_daily would be sat next to one another rather than dim_users and fct_users_daily.

A drawback is that it’s less clear what type of table it is. I personally prefer this approach.

File you need to create (feel free to use dim_orders or orders here!):

models/marts/orders.sql

WITH

-- Aggregate measures
order_item_measures AS (
	SELECT
		order_id,
		SUM(item_sale_price) AS total_sale_price,
		SUM(product_cost) AS total_product_cost,
		SUM(item_profit) AS total_profit,
		SUM(item_discount) AS total_discount

	FROM {{ ref('int_ecommerce__order_items_products') }}
	GROUP BY 1
)

SELECT
	-- Dimensions from our staging orders table
	od.order_id,
	od.created_at AS order_created_at,
	od.shipped_at AS order_shipped_at,
	od.delivered_at AS order_delivered_at,
	od.returned_at AS order_returned_at,
	od.status AS order_status,
	od.num_items_ordered,

	-- Metrics on an order level
	om.total_sale_price,
	om.total_product_cost,
	om.total_profit,
	om.total_discount

FROM {{ ref('stg_ecommerce__orders') }} AS od
LEFT JOIN order_item_measures AS om
	ON od.order_id = om.order_id
